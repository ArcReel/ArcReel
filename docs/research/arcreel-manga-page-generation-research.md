**调研日期**：2026-07-19
**用途**：评估用 ArcReel 已集成的图像后端生成"带中文对话气泡的多格漫画页"三条候选技术路线的可行性，作为后续 PoC / 设计文档的输入素材
**币种**：各供应商定价保持原始币种（CNY / USD），不做汇率换算——跨币种折算涉及浮动汇率假设，超出"已核实供应商数据"范畴，本报告只在同币种内比较
**定价数据来源**：全部取自 `lib/config/registry.py`（`PROVIDER_REGISTRY`），即 ArcReel 当前实际计费口径；与官方公开定价页存在差异处已单独标注

**信息来源与可信度声明**：
- 本地能力基线：直接阅读 `lib/image_backends/`、`lib/grid/`、`lib/prompt_builders.py`、`lib/config/registry.py`、`lib/pricing/types.py` 源码，逐条给出文件路径 + 行号引用。
- 外部能力核验：对 8 个候选模型（Ark Seedream 系列、Google Gemini、OpenAI gpt-image-2、DashScope Qwen-Image、xAI Grok Imagine、MiniMax image-01、Kling、Vidu）各派发一次独立调研，优先级为「官方一手文档/API 参数表 > 官方博客/Cookbook > 第三方独立评测机构（如 Artificial Analysis Elo 榜） > 第三方 API 中转商文档（fal.ai / apimodels.app 等） > 中文社区实测（知乎/智东西/B 站等） > 营销性博客/公关通稿」。每条能力声明均标注来源类型、URL（如适用）与访问日期 2026-07-19；无法核实的一律标注「证据不足/存疑」，不采用训练记忆猜测。
- 部分官方站点为 SPA，WebFetch 直接抓取返回空壳或 403/446，改用真实渲染方式重新提取正文，详见各模型小节脚注。

## 0. 调研范围与定位

**评估目标**：ArcReel 当前 8 个已注册图像模型是否可以支撑「一页多格、格内含中文对话气泡的漫画页」生成需求，以及应该选择哪条技术路线落地。

**三条候选路线**：
- **(a) 整页直出**：prompt 描述分格布局与对话文字，模型一次调用输出整页成品（含中文文字）。
- **(b) 逐格生成 + 程序拼版**：每格独立调用生成，程序化组合成页（版式由代码控制，格内是否含文字取决于实现）。
- **(c) 无字页 + 程序化叠字**：模型只产出分格插画（无气泡/空气泡），对话文字完全由后期程序化贴字排版加入。

**六个评估维度**：分格构图可控性、中文对话可靠性、跨格/跨页角色一致性、按页成本、失败率/重试先验、与现有架构的集成难度。

**不在范围内**（属后续 PoC / 设计文档阶段决策）：具体字体/排版引擎选型、气泡自动布局算法细节、前端 UI 改动、新增供应商评估、成本熔断规则。

## 1. ArcReel 本地能力基线

### 1.1 图像后端抽象

`lib/image_backends/` 下各供应商后端统一实现 `ImageBackend` Protocol，围绕 `ImageCapability`（`TEXT_TO_IMAGE` / `IMAGE_TO_IMAGE`）、`ImageGenerationRequest` / `ImageGenerationResult` 展开。参考图缺失/不可读时统一 fail-loud（`ImageCapabilityError` + `image_reference_images_unreadable`），Kling（`lib/image_backends/kling.py:140-144`）、MiniMax 均有此实现，跨供应商行为一致，任何新增漫画生成路径都可直接复用这一契约，无需新设计错误处理。

### 1.2 Prompt 单一真相源与反文字约束——路线 (a) 的直接架构冲突

`lib/prompt_builders.py` 是分镜 / 资产图 prompt 的唯一出口。第 42 行声明：

```python
_NEGATIVE_TAIL_ASSET = "画面避免：水印、多余文字、Logo。"
```

`append_image_negative_tail()`（`lib/prompt_builders.py:151-161`）对所有经归一化的分镜图 prompt **无条件**追加这条反向提示词。这意味着 ArcReel 现有分镜图生成管线的默认行为是"禁止模型画文字"，路线 (a)（整页直出，需要模型渲染中文对话）若要落地，必须新增一条与此完全相反的例外路径——绕开这一默认行为本身就是一处不小的架构改动，且需要判断该例外只对"漫画页"场景生效、不能误伤其他分镜图生成。

`lib/prompt_builders.py:124-148` 的 `append_product_fidelity_tail()` 提供了"在既有 prompt 上追加强指令"的先例（追加产品高保真还原指令，幂等、可重复调用），若路线 (a) 或 (b) 需要引导模型"预留气泡空白区域"，可参考此模式新增专用 tail 函数，而非另起一套 prompt 拼接逻辑。

### 1.3 宫格图系统——路线 (b)/(c) 的可复用基础与缺口

`lib/grid/` 是 ArcReel 现有的"宫格生视频"系统，与本次调研的多格漫画页需求在架构哲学上高度相关，值得重点比对：

- **`lib/grid/layout.py`**：`GridLayout` frozen dataclass + `_GRID_CONFIGS` 静态表 + `calculate_grid_layout()`（第 54 行）——版式（几行几列、每格宽高比）由**代码确定性计算**，不依赖模型自行判断。这一设计本身就隐含了一个判断：ArcReel 团队此前在"是否信任模型一次性渲染精确版式"这一问题上，选择了不信任，转而用程序化布局。
- **`lib/grid/prompt_builder.py::build_grid_prompt()`**：第 162-164 行显式声明负面约束：
  ```
  【负面约束】
  禁止出现以下任何元素：
  - 文字、字幕、标签、标题、数字编号、时间戳
  ```
  这是与 `_NEGATIVE_TAIL_ASSET` **相互独立的第二处**反文字约束，进一步印证 ArcReel 现有生成管线的默认策略是"模型只管画面内容，一切文字/标签交给确定性代码或后续环节处理"。这一既有策略与路线 (c) 完全兼容，与路线 (a) 直接冲突。
- **`lib/grid/splitter.py`**：仅含 `split_grid_image()`、`center_crop_to_ratio()`、`is_placeholder_cell()` 三个函数，**全部是"拆分"方向**（把模型生成的整张宫格图切成若干格用于视频首尾帧），**没有反向的"组装/合成"函数**。若路线 (b)/(c) 采用"逐格生成后拼版成页"，需要新建一个组装模块（如 `assemble_page()`），现有代码不能直接复用，只能复用其"确定性版式计算"的设计思路（`layout.py`）。

### 1.4 集成范式差异

Kling（`lib/image_backends/kling.py`）走 submit/poll 异步模式（`_POLL_INTERVAL_SECONDS=5.0`、`_POLL_MAX_WAIT_SECONDS=600.0`，见第 54-55 行），MiniMax 走同步单步模式。这一差异影响「失败率/重试先验」维度下的实际重试延迟成本：异步供应商单格失败后的重试摩擦（等待轮询超时）高于同步供应商，是路线 (b)/(c) 选择"哪家供应商做逐格生成"时的次要考量。

## 2. 八个候选模型逐一能力核验

> 以下每条能力声明标注来源类型与访问日期；无法找到可靠证据的一律标注「证据不足/存疑」。

### 2.1 Ark Seedream（4.0 / 4.5 / 5.0 / 5.0-lite）

- 官方口径的中文文字渲染量化数据仅覆盖 4.0→4.5 迭代：准确率从 38% 提升到 61%（修复后 74%）（来源：知乎《Seedream 4.5 三大能力拆解实测》，访问 2026-07-19，第三方转述官方/评测数字，中等可信度）。
- **5.0 / 5.0-lite 无官方文字渲染数据**，不可假设其继承或优于 4.5 的表现。
- 官方/社区确认支持"连环画/组图/分镜故事板"生成，但这是"同批次生成多张独立图片"，**没有证据表明模型能一次性输出整页多格拼版 + 对话气泡的复合版式**。
- **模型身份存疑**：第三方站 apimodels.app 将 `doubao-seedream-5-0-260128` 标注为"Doubao Seedream 5.0 Lite"，与 ArcReel 后端"5.0 与 5.0-lite 是两个独立模型"的假设存在冲突，未获官方一手交叉确认；registry.py 中两者定价恰好都是 ¥0.22/张（`lib/config/registry.py:522,528`），进一步加深该存疑，建议上线前用现有 API Key 直接实测核实。
- 参考图：4.0/4.5 官方 2-10 张级别，受"参考图+生成图 ≤ 15"总量约束；5.0/5.0-lite 参考图上限证据薄弱（第三方称"up to 5"，未获官方确认）。
- 定价（`_ark_image_pricing`，CNY，本地镜像 `docs/ark-docs/火山方舟费用参考.md` 交叉确认一致，高置信度）：4.0 ¥0.20/张、4.5 ¥0.25/张、5.0 ¥0.22/张、5.0-lite ¥0.22/张。

### 2.2 Google Gemini（gemini-3-pro-image-preview / gemini-3.1-flash-image-preview）

- 官方教程页（ai.google.dev/gemini-api/docs/image-generation，访问 2026-07-19）设有专门章节"Sequential art (Comic panel / Storyboard)"，提供官方提示词模板"Make a 3 panel comic..."——是本次调研中少数**官方文档直接给出多格漫画 prompt 模板**的模型（高置信度，官方一手）。
- **关键限制**：第三方实测（anifusion.ai，访问 2026-07-19）明确指出该模型"is optimized for full-page generation... rather than multiple smaller panels"——即模型倾向渲染单一连贯画面，而非工整分割、带清晰边框的多格版式。官方模板存在，不代表模型能稳定输出"整齐分格 + 边框 + 独立气泡"的漫画页构图，这一发现直接影响路线 (a) 的可行性判断。
- gemini-3-pro-image-preview 官方明确支持最多 **14 张参考图**（6 物体高保真 + 5 人物一致性），gemini-3.1-flash-image-preview 对应上限未获一手确认（证据不足）。
- 中文（zh-CN）在官方"最佳表现语言"清单内；3.1-flash 官方页面提及"Improved i18n text rendering"，但**无中文量化准确率官方数字**（第三方给出的 90%+ 数字均为非官方自测，不采信）。
- 定价（`_gemini_image_pricing`，USD，`lib/config/registry.py:368,376-377`）：
  - gemini-3-pro-image-preview：1K/2K $0.134，4K $0.24
  - gemini-3.1-flash-image-preview：512px $0.045，1K $0.067，2K $0.101，4K $0.151

### 2.3 OpenAI gpt-image-2

- 官方 Cookbook 有专门"4.7 Story-to-Comic Strip"章节，是本次调研找到的**最直接的官方多格漫画支持证据**（高置信度）。
- 官方 2026-04-21/22 正式发布，命名与 ArcReel 一致；定价交叉验证：官方 (quality, size) 静态兜底表数字与 registry.py 完全一致（`lib/config/registry.py:818-827`，1024×1024 low=$0.006/medium=$0.053/high=$0.211，token 计价 image_in $8/M、image_out $30/M 亦一致，高置信度交叉验证）。
- 中文/CJK 文字渲染准确率数字（90%/99%）**均来自第三方/自媒体转述，非官方发布**，需标注存疑。

- LM Arena 2026-04-21 gpt-image-2 登顶 Text-to-Image 榜（Elo 1512，领先第二名 242 分，第三方独立评测，高可信度，侧面印证综合生成质量但非漫画专项）。
- 定价（`PerImageOpenAIToken`，USD）：token 主路径 image_in $8/M、image_cached_in $2/M、image_out $30/M、text_in $5/M；(quality,size) 兜底见上。

### 2.4 DashScope Qwen-Image（2.0 / 2.0-pro）

- **关键正面发现**：官方博客（qwen.ai，2026-02-07，访问 2026-07-19）展示了单次调用生成"4×6=24 格漫画"（含格内中文对话气泡，具体文本"终于完成了！生态球"）的官方案例——是本次调研发现的**唯一直接证明"单页多格 + 中文气泡一次直出"可行的官方证据**（高置信度来源，但仅出现在营销博客，非 DashScope 正式 API 能力清单条款，属孤例而非批量统计的成功率数据）。
- 长文本渲染：官方《兰亭集序》全文渲染案例 + 提示词 1300 token 上限（官方 API 文档确认，高置信度）。
- 参考图 1-3 张，与本地 `_QWEN_REF_LIMIT = 3` 完全吻合（官方一手确认）。
- 分辨率 512×512-2048×2048（官方仅约束总像素，未显式限定宽高比范围）。
- 社区测评（二手转述，可信度中等）反馈"角色数据不足、需详细人设、分镜有提升空间"。
- 定价（`_dashscope_image_pricing`，CNY，`lib/config/registry.py:970,977`）：2.0 ¥0.2/张、2.0-pro ¥0.5/张——**按成功生成的图片数计费，与格数无关**，24 格拼版案例若可复现，单页成本仅 ¥0.2。

### 2.5 xAI Grok Imagine（grok-imagine-image / grok-imagine-image-pro）

- **模型命名/计费漂移存疑**：当前 docs.x.ai 只列 grok-imagine-image($0.02/张) 和 grok-imagine-image-quality($0.05/张)，未见 grok-imagine-image-pro；第三方称该 pro 版已于 2026-05-15 弃用迁移至 quality 版。而 ArcReel registry.py 当前仍注册 `grok-imagine-image-pro` 且定价为 $0.07/张（`lib/config/registry.py:733-738`），与官方当前目录及第三方转述的迁移价格均不一致——**需人工核实该模型 ID 当前是否仍可正常调用**，本报告不代为判断，仅如实标注证据冲突。
- Atlas Cloud 第三方实测（2026-05-15，有明确方法论）：简体"时光电影节"被渲染为繁体"時光電影節"，评测中列为"hard disqualifier"（硬性失格）——是本次调研中最具体的中文渲染负面证据。
- 官方/文档均无多格漫画/故事板支持声明；社区共识是"单格出图，多格需应用层拼接"。
- 参考图数量文档间不一致（本地镜像 5 张 vs 官方当前文档 3 张，随模型代次波动，证据不足）。
- 分辨率仅 1K/2K，最极端竖版比例约 9:20，不足以支撑高密度整页竖版漫画。
- 定价（`_grok_image_pricing`，USD，`lib/config/registry.py:738,745`）：grok-imagine-image-pro $0.07/张（存疑，见上）、grok-imagine-image $0.02/张。

### 2.6 MiniMax image-01

- 官方文档/新闻稿**均未提及**中文文字渲染能力或多格漫画场景。
- 第三方实测（302.AI 知乎测评，页面 403 无法直接抓取，仅获搜索摘要）：清明节主题测试图片右下角出现无法识别字符——负面信号，未成体系。

- 定价（`_minimax_image_pricing`，CNY，`lib/config/registry.py:1107`）：¥0.025/张（成功才扣费）——8 个候选模型中单价最低。

### 2.7 Kling（kling-image-o1 / kling-v3-omni-image）

- 官方 API 参数文档（klingai.com/document-api/api/image/o1/image-generation，访问 2026-07-19）确认：`image_list`（参考图）+ `element_list`（主体库主体）总数 ≤ 10，与本地 `_MODEL_REF_LIMITS["kling-image-o1"]=10`（`lib/image_backends/kling.py:60`）完全一致；第三方 fal.ai 模型页独立佐证同一数字（高置信度）。
- **官方"漫画/连载"定位需要纠偏**：官方用户指南（kling.ai/quickstart/klingai-image-o1-user-guide，2025-12-30 发布）全文仅两处提及"漫画"相关表述，且均**不涉及整页多格 + 对话文字排版**：一处是"ideal for IP character design, comic scene creation..."（跨图角色一致性场景下的概括性描述，指单图/单场景，非成页版式）；另一处是"漫画风格滤镜"（把整张图转换成漫画/复古连环画/美漫风格，本质是画风转换，不是版面生成）。第三方聚合站 MindStudio 页面转述的"serialized visual content"（连载内容）一词**未在官方原文中逐字出现**，属第三方转述放大，不宜等同官方承诺。
- **官方无"整页多格漫画 + 对话气泡"案例**：对官方用户指南全文（约 1100 行）检索"panel/dialogue/bubble/caption/分格/气泡/台词"关键词均无命中。官方"灵动画布"（Kling Canvas）提供"故事分镜"模板，但产出是**一组独立分镜图像供用户手动排布**，并非自动合成为单张带边框分格 + 嵌入对话气泡的成品漫画页——这是本次 8 个模型核验中**唯一获得官方文档"穷尽关键词检索确认不支持"级别否定证据**的模型，可信度高（否定性结论）。
- 中文文字渲染能力：官方未文档化；第三方评测侧面负面（"Kling O1 Image matches Flux 2 in quality/speed but Flux 2 wins in realism and text rendering"，即 Kling 图像模型文字渲染弱于 Flux 2）；社区无 kling-image-o1（图像模型，区别于同名视频模型）中文文字渲染的直接实测证据。
- 分辨率官方最高支持 4K（ArcReel registry.py 当前仅声明 1K/2K 两档用于计费展示，存在口径差异，非本次调研取舍范畴，如实记录）；竖版最大比例 9:16，不支持更极端的条漫长图比例（第三方 MindStudio 页面提及的"1:9"与官方 API 枚举列表矛盾，判定为第三方转述有误，不采用）。
- 定价（`lib/config/registry.py:1189,1199`，CNY）：kling-image-o1 ¥0.2/张（flat）；kling-v3-omni-image 1K/2K ¥0.2/张、4K ¥0.4/张。

### 2.8 Vidu（viduq1 / viduq2）

- 官方定价文档（platform.vidu.cn/docs/pricing，访问 2026-07-19，高置信度一手数据）确认：**1 credit = ¥0.03125**（国内站），**1 credit = $0.005**（国际站）。`reference2image` 按参考图数量（1-3 / 4-7）与分辨率（1080p/2K/4K）分档计费，viduq1 固定 20 credits/次（约 ¥0.625/次）。
- 中文文字渲染**官方与第三方均无证据**；业内横评甚至将"中文文字完美渲染"归给竞品"即梦"而非 Vidu，本模型在该维度证据最薄弱。
- 官方无"多格漫画"命名功能。第三方独立评测机构 Artificial Analysis（高可信度，直接 WebFetch 榜单页面）：Vidu Q2 在 Image Editing 榜排名第 45（Elo 1130）、Text-to-Image 榜第 64（Elo 1099），与中文媒体"超越 GPT-5、并列前三"的宣传口径明显不符，应视为公关话术，不采信。
- 参考图 7 张上限确认，2:3 比例仅 viduq2 支持，均与本地基线一致；无更极端竖版长图比例证据。
- 定价类型为 `ViduDelegate`（`lib/pricing/types.py:155-164`），实际费率委托 `lib.vidu_shared.calculate_vidu_cost` 按 credits 结算，非固定 PerImageFlat。

## 4. 三条候选路线综合评估

### 4.1 路线 (a)：整页直出

| 维度 | 评估 |
|---|---|
| 分格构图可控性 | **低**。8 个模型均未提供"精确 N 格网格 + 边框位置"的 API 级控制参数，完全依赖 prompt 自然语言描述。第三方实测明确指出 Gemini "倾向整页连贯画面而非清晰分割多格"；Kling 官方文档穷尽关键词检索确认无此功能。仅 Qwen-Image 有一例官方博客案例证明 24 格网格可行，但为孤例，未见批量统计的成功率数据。 |
| 中文对话可靠性 | **两极分化**。Qwen-Image 有官方案例 + 长文渲染证据支撑；gpt-image-2 有官方 Story-to-Comic 章节但中文准确率数据均为第三方转述；其余 6 家（Ark 5.0 系 / Gemini / Grok / MiniMax / Kling / Vidu）证据薄弱、空白或明确负面（Grok 简繁混淆判定"硬性失格"）。 |
| 跨格/跨页角色一致性 | **格内天然一致**（同一画布单次生成），是本路线唯一显著优势；跨页仍需依赖参考图机制。 |
| 按页成本 | **最优**，1 次调用，如 Qwen-Image 2.0 仅 ¥0.2/页。 |
| 失败率/重试先验 | **最差**。整页任一格出错（字符错位、格子缺失、气泡位置错误）等同整页作废，需整页重新生成；没有任何供应商发布过"整页多格+文字"场景的失败率数据，纯属未知。 |
| 集成难度 | **高**。需新增绕开 `append_image_negative_tail()`（`lib/prompt_builders.py:151-161`）的例外 prompt 路径，且需要新建"整页校验器"（校验格数/文字/气泡位置是否达标）——项目内无同类先例。 |

### 4.2 路线 (b)：逐格生成 + 程序拼版

| 维度 | 评估 |
|---|---|
| 分格构图可控性 | **高**。可直接复用 `lib/grid/layout.py` 的确定性版式计算模式（`_GRID_CONFIGS` 静态表 + `calculate_grid_layout()`），程序控制每格位置/大小，模型只需产出格内画面。 |
| 中文对话可靠性 | **视文字策略而定**。若各格仍由模型自行渲染对话文字，则继承路线 (a) 同等的中文渲染不确定性；若各格也走"无字"策略，则退化为与路线 (c) 相同的可靠性水平。 |
| 跨格/跨页角色一致性 | **中**。依赖各供应商多参考图/角色一致性机制（Kling 1-10 图、Gemini 14 图、gpt-image-2 约 16 张均有官方文档支撑），但均针对"角色设计一致性"验证，未见针对"连续剧情分格"场景的实测数据，需 PoC 验证。 |
| 按页成本 | **中**，4-6 次调用，如 Ark 4.5 四格 ¥1.00/六格 ¥1.50、Qwen-Image 2.0 四格 ¥0.8/六格 ¥1.2，仍在低成本区间。 |
| 失败率/重试先验 | **优于 (a)**。单格失败只需重试单格，颗粒度对齐现有 `GenerationQueue` 的按任务重试机制，不必整页重来。 |
| 集成难度 | **中**。需新建程序化拼版模块（`lib/grid/splitter.py` 目前只有 `split_grid_image()`/`center_crop_to_ratio()`/`is_placeholder_cell()` 三个拆分函数，无反向合成函数），但可复用现有 `GenerationQueue` 的多任务编排与参考图注入机制。 |

### 4.3 路线 (c)：无字页 + 程序化叠字

| 维度 | 评估 |
|---|---|
| 分格构图可控性 | **高**，同 (b)，程序控制版式。 |
| 中文对话可靠性 | **最高**。完全绕开"模型渲染中文"这一在 8 个模型中证据最参差不齐的能力项，与 ArcReel 现有 `_NEGATIVE_TAIL_ASSET`（`lib/prompt_builders.py:42`）及 `build_grid_prompt()` 负面约束（`lib/grid/prompt_builder.py:162-164`，本就禁止模型自绘文字）天然兼容，零架构冲突。 |
| 跨格/跨页角色一致性 | **中**，同 (b)，依赖各供应商参考图机制，需 PoC 验证。 |
| 按页成本 | **中**，与 (b) 相同（每格 1 次调用，价格相同），但省去"整页/单格文字校验失败重试"的隐藏成本。 |
| 失败率/重试先验 | **最优**。文字环节完全由确定性代码控制，不存在"模型没画对字"的失败模式，失败率来源只剩画面内容质量本身。 |
| 集成难度 | **中偏高**。除 (b) 的拼版需求外，还需新建对话气泡自动布局引擎（气泡形状/位置、竖排/横排、字体、气泡内换行），是纯本地图像处理工作，不依赖任何供应商能力，但工作量本身不小。 |

## 5. 结论：收窄到 2 条候选路线进入 PoC

### 淘汰：路线 (a) 整页直出

理由：
1. 8 个模型中仅 Qwen-Image（官方博客孤例）与 gpt-image-2（官方 Cookbook 章节，但中文数据为第三方转述）提供直接正面证据；其余 6 家证据薄弱、空白或明确负面，其中 Kling 是本次调研中唯一获得"官方文档穷尽关键词检索确认不支持"级别否定证据的模型。
2. 分格构图可控性天然弱于程序化方案——ArcReel 自有的宫格视频系统（`lib/grid/`）本身已经选择了"确定性版式计算 + 模型仅填格内内容"的架构，并在 prompt 中明确禁止模型自绘文字/编号，这一既有设计已隐含判断"依赖模型一次性渲染精确版式"风险过高。
3. 与现有架构冲突最大：`append_image_negative_tail()` 无条件为所有分镜图追加反文字约束，路线 (a) 需要新增一条例外路径，且中文文字渲染本身仅 1-2 家模型有可信正面证据，风险与收益不成比例。
4. 按页成本优势（1 次调用）不足以抵消可控性与可靠性缺口——整页失败等同全页重新生成，而非颗粒度更细的单格重试。

### 保留：路线 (b) 逐格生成 + 程序拼版 与 路线 (c) 无字页 + 程序化叠字

理由：两者共享同一套底层机制（逐格独立生成 + 程序化版式控制），与 ArcReel 现有 `lib/grid/` 的架构哲学高度一致，可复用其确定性版式计算思路而非另起炉灶。鉴于 8 个模型中中文文字渲染证据薄弱或负面占多数（Grok 简繁混淆判定"硬性失格"，Kling/MiniMax 官方未文档化且第三方评测指出 Kling 弱于 Flux 2），建议将 **(c) 作为 (b) 的推荐落地形态**：模型只画无字画面，对话文字交给确定性的程序化排版，把整条路线的可靠性下限从"最弱一环的模型中文渲染能力"提升到"ArcReel 自己可控的文字排版代码"——这是唯一能让"中文对话可靠性"这一维度脱离"看运气"状态的路线。

### PoC 阶段需要验证的关键假设

1. **跨格角色一致性的实际表现**：现有证据均针对"角色设计图/单主体"场景，未见针对"同一角色连续出现在 4-6 个无字分格中"的实测数据。PoC 需要用 1-2 家候选供应商（建议 Ark Seedream 4.5、Qwen-Image 2.0，中文场景成本最低且证据相对完整）实际产出连续剧情分格，人工评审一致性。
2. **程序化拼版模块的实现复杂度**：`lib/grid/splitter.py` 目前只有拆分函数，无组装函数，需新建 `assemble_page()` 一类模块。PoC 需验证是否只需支持规则等宽网格，还是需要支持漫画常见的非对称/跨格版式（这会显著提高实现复杂度）。
3. **程序化对话气泡叠字方案的实际观感**：气泡形状自动贴合人物位置、竖排/横排选择、气泡内自动换行、字体选型——这是纯本地图像处理问题，不依赖任何供应商能力，PoC 需要验证效果是否达到"像漫画"而非"贴标签感"。
4. **是否需要引导模型预留气泡空白区域**：若叠字观感依赖画面中存在合适的留白位置，需要验证 prompt 能否稳定引导模型"预留气泡形状空白"（可参考 `append_product_fidelity_tail()` 的"追加强指令"先例新增专用 tail），建议挑选 1-2 家候选供应商做实测。
5. **逐格生成的实际失败率**：目前证据均来自静态文档，没有任何供应商发布过批量生成场景的实测失败率。PoC 需要跑真实的 4-6 格 batch，观察实际重试频率与延迟成本（尤其 Kling 等异步 submit/poll 模式的重试摩擦）。
6. **Ark Seedream 5.0/5.0-lite 模型身份存疑**、**Grok imagine-image-pro 当前可用性存疑**：这两项与本报告核心结论无关，但若 PoC 选中相关模型作为候选，须先行核实，避免在错误的模型身份假设上投入实现成本。
